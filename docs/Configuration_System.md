# Configuration_System.md — Smart Retail OS Configuration Reference

**Depends on:** PRD.md, Database.md, Security.md, AI.md, Sync_Architecture.md, Hardware.md, Notifications.md, Design_System.md, Coding_Standards.md
**Feeds into:** UI_UX.md §2.6 (Settings & Administration screens), Implementation_Pipeline.md (each setting's owning Stage)
**Governing rule:** Every setting is data-driven (stored in `companies`/`branches`/`subscription_plans`/user preference tables), never a hardcoded per-tier `if` in application code (Database.md §2.16 entitlement design philosophy extended to configuration generally). **Scope** below indicates the level at which a setting is stored/overridable: Company, Branch, Device, User, or Platform (Platform = Platform-Admin-only, never tenant-visible).

## 1. Setting Documentation Format

For every setting: **Name**, **Description**, **Type**, **Allowed Values**, **Default Value**, **Validation Rules**, **Dependencies**, **Restart Required**, **Scope**.

## 20. Configuration Precedence Rules

- **Company → Branch → Device → User**, in that order of increasing specificity, wherever a setting exists at multiple scopes — a more specific scope's explicit value overrides the broader scope's default, but only where the broader-scope setting's own definition permits an override (mirrors the "inherited vs. overridden per field" model in Notifications.md §1 and Sync_Architecture.md §3.3 for settings entities).
- **Platform-scope settings are never tenant-editable** in v1 — they represent product-wide policy (trial length, MFA mandate, log redaction) or implementation-tunable constants not yet exposed as business configuration. Exposing any of these as tenant-configurable in a later version is a PRD change, not a configuration-system change.
- A setting change is itself a Class B sync entity (per Sync_Architecture.md §3.3) unless explicitly marked Platform-scope (which lives server-side only and is not part of the tenant sync stream).
- No setting listed here may be used to bypass a rule in Business_Rules.md or a hard limit in Security.md (e.g., `security.mfa.platformAdminRequired` cannot be set to `false` under any configuration path) — such settings are documented as **locked** specifically to make this non-negotiability explicit rather than silently unenforced.
