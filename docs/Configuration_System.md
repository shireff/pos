## 20. Configuration Precedence Rules

- **Company â†’ Branch â†’ Device â†’ User**, in that order of increasing specificity, wherever a setting exists at multiple scopes â€” a more specific scope's explicit value overrides the broader scope's default, but only where the broader-scope setting's own definition permits an override (mirrors the "inherited vs. overridden per field" model in Notifications.md Â§1 and Sync_Architecture.md Â§3.3 for settings entities).
- **Platform-scope settings are never tenant-editable** in v1 â€” they represent product-wide policy (trial length, MFA mandate, log redaction) or implementation-tunable constants not yet exposed as business configuration. Exposing any of these as tenant-configurable in a later version is a PRD change, not a configuration-system change.
- A setting change is itself a Class B sync entity (per Sync_Architecture.md Â§3.3) unless explicitly marked Platform-scope (which lives server-side only and is not part of the tenant sync stream).
- No setting listed here may be used to bypass a rule in Business_Rules.md or a hard limit in Security.md (e.g., `security.mfa.platformAdminRequired` cannot be set to `false` under any configuration path) â€” such settings are documented as **locked** specifically to make this non-negotiability explicit rather than silently unenforced.
